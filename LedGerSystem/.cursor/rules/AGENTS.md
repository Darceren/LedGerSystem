# AI Coding Rules

## 核心原则

### 1. 先思考，再编程
在输出代码前：

- 先分析需求
- 先理解现有代码结构
- 先列出实现步骤
- 明确可能的风险点
- 不要直接开始写代码

必须先输出：
1. 需求理解
2. 实现方案
3. 风险点
4. 修改范围

确认逻辑后再写代码。

---

## 2. 简单优先

优先选择：

- 最简单方案
- 最少代码方案
- 最容易维护方案
- 最容易阅读方案

禁止：

- 过度设计
- 过度抽象
- 不必要设计模式
- 不必要泛型
- 不必要继承

---

## 3. 不破坏现有业务

修改代码时：

- 优先兼容旧逻辑
- 不随意修改接口结构
- 不随意修改返回字段
- 不随意改数据库字段
- 不随意改变业务流程

如果必须修改：

必须说明：
- 为什么要改
- 会影响什么
- 如何兼容

---

## 4. 优先复用已有代码

写新代码前：

必须先检查：
- 是否已有类似方法
- 是否已有工具类
- 是否已有公共逻辑
- 是否已有统一返回结构

禁止重复造轮子。

---

## 5. 注意性能

必须考虑：

- SQL 性能
- 循环性能
- 内存占用
- 网络请求次数
- 锁竞争
- 大数据量情况

避免：

- N+1 查询
- 多次重复查询数据库
- 不必要 ToList()
- 不必要 Count()
- 不必要字符串拼接

---

## 6. 注意空值问题

所有代码必须考虑：

- null
- 空字符串
- 空集合
- DBNull
- 请求参数缺失

避免：

- NullReferenceException
- IndexOutOfRangeException

---

## 7. 注意并发安全

涉及：

- Redis
- 缓存
- 数据库
- 计数器
- 余额
- 订单

时必须考虑：

- 并发问题
- 重复提交
- 幂等性
- 锁机制

---

## 8. API 编写规范

API 必须：

- 使用 async/await
- 支持取消 Token
- 返回统一格式
- 正确处理状态码
- 输出必要日志
- 处理异常

---

## 9. SQL 规范

SQL 必须：

- 明确索引使用
- 避免 SELECT *
- 注意事务范围
- 注意锁问题
- 注意死锁风险

优先：

- EXISTS
- TOP
- LIMIT
- 索引字段查询

---

## 10. 日志规范

必须记录：

- 错误日志
- 第三方请求异常
- 关键业务异常
- 转发失败
- 支付失败

日志必须包含：

- 时间
- 请求参数
- 异常信息
- 关键业务ID

---

## 11. 网关/转发项目特殊规则

对于转发请求：

必须注意：

- Request.Body 是否已被读取
- 是否 EnableBuffering()
- 是否复位 Position = 0
- Header 是否完整转发
- QueryString 是否完整
- ContentType 是否正确
- Response 是否已开始写入

禁止：

- 提前消费 Body 后不复位
- 丢失 Header
- 丢失 Content-Type
- 返回格式不一致

---

## 12. 输出代码要求

输出代码时：

- 给完整代码
- 不省略关键逻辑
- 标注修改位置
- 给必要注释
- 说明为什么这样实现

如果有更优方案：

先说明，再给代码。

---

## 13. Debug 原则

出现错误时：

先分析根因：

- 请求问题？
- Content-Type？
- Body 被消费？
- 路由问题？
- 中间件问题？
- 并发问题？
- SQL问题？

不要直接“猜”。

---

## 14. 安全规范

禁止：

- 硬编码密钥
- 明文密码
- SQL 注入风险
- 未验证输入
- 信任客户端数据

必须：

- 参数校验
- 输入过滤
- 安全日志处理

---

## 15. AI 输出风格

输出顺序：

1. 需求分析
2. 风险分析
3. 实现方案
4. 完整代码
5. 注意事项

不要直接只给代码。

## 16. 测试规范

### 1. 修改后必须说明如何测试

每次修改代码后，必须给出测试方式，包括：

- 正常场景
- 异常场景
- 边界场景
- 并发场景
- 回归测试范围

不要只给代码，不说明怎么验证。

---

### 2. API 测试要求

涉及 API 时，必须提供：

- 请求地址
- 请求方式
- Header
- Query 参数
- Body 示例
- 预期返回结果

如适合，提供 Postman / curl 示例。

---

### 3. 网关转发测试要求

涉及请求转发时，必须测试：

- QueryString 是否完整转发
- Header 是否完整转发
- Body 是否完整转发
- Content-Type 是否正确
- 上游状态码是否原样返回
- 上游 Response Header 是否正确返回
- 上游返回空 Body 是否正常
- 上游超时是否有日志
- Request.Body 被读取后是否已 Position = 0

---

### 4. 支付/订单测试要求

涉及支付、订单、钱包、余额时，必须测试：

- 重复回调
- 重复提交
- 金额精度
- 订单不存在
- 订单已处理
- 金额不匹配
- 并发处理同一订单
- 第三方接口超时
- 第三方返回异常

必须考虑幂等性。

---

### 5. SQL 测试要求

涉及 SQL / ORM 查询时，必须测试：

- 空数据
- 单条数据
- 多条数据
- 大数据量
- 分页边界
- 排序正确性
- 查询条件为空
- 查询条件组合
- 执行计划是否合理
- 是否使用索引

禁止只在少量测试数据下验证。

---

### 6. 前端测试要求

涉及前端页面时，必须测试：

- PC 浏览器
- 手机浏览器
- 空数据
- 长文本
- 网络失败
- 重复点击
- 按钮禁用状态
- 金额/数字显示
- 异常提示
- 兼容旧浏览器能力

---

### 7. 回归测试要求

修改已有功能时，必须说明：

- 影响了哪些旧功能
- 哪些接口需要重新测试
- 哪些页面需要重新测试
- 是否影响数据库结构
- 是否影响第三方接口

---
